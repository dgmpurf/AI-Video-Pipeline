## 7. Identifier Grammar, Ordering, And Occurrence Evidence

`validate_all_identifier_occurrences(stdout, stderr)` implements the exact order:

1. stdout, then stderr;
2. full-stream strict JSON;
3. each CRLF/CR/LF-delimited nonempty line as strict JSON in ascending line order;
4. complete-channel text tokens left to right.

JSON objects are traversed depth-first preorder, dictionary members in parser source order, and arrays by ascending index. Each member is inspected before recursion into its value.

Structured keys use surrounding ASCII-whitespace trimming, ASCII-only lowercase conversion, and underscore/hyphen removal. The only normalized keys are submitid and logid. Structured values are strings or integers excluding booleans, are ASCII-trimmed, and must fully match `[A-Za-z0-9._:-]+`.

Text keys accept exactly submit_id, submit-id, submitid, logid, log_id, and log-id, case-insensitively, unquoted or enclosed in matching single/double quotes. Separators are colon, equals, or one-or-more ASCII whitespace. Values use the same token alphabet, require a complete delimiter, and cannot begin or end inside a longer ASCII identifier.

Each occurrence records:

- identifier_type and case-preserved value;
- stdout/stderr channel;
- full_stream_json, line_json, or text_token source kind;
- frame index;
- applicable line number;
- character offset in the channel;
- applicable JSON path;
- globally monotonic one-based discovery order.

Full-stream frame_index is 0, line-JSON frame_index is its one-based physical line number, and text-token frame_index is its zero-based match ordinal. Duplicate-equal occurrences from different source families are preserved.

Submit IDs and logids are classified independently as MISSING, UNIQUE, DUPLICATE_EQUAL, or CONFLICTING. Only UNIQUE and DUPLICATE_EQUAL select the first value. Any distinct value conflict, duplicate JSON key, nonfinite JSON frame, or invalid structured identifier value sets contradictory_identifiers_present=true and safe-blocks selection as required.
