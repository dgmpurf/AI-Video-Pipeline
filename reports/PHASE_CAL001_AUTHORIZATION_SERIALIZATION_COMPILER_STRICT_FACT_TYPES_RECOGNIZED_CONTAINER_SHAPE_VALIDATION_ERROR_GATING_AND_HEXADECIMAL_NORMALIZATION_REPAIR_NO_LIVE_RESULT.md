## 6. Root Causes B1 Through B4

### B1: loose JSON fact typing

Selected facts were compared with Python equality. Because `1 == True` and `0 == False`, malformed integer/boolean representations could match the derived fact. A valid selected occurrence could also hide a malformed secondary occurrence, while identical malformed duplicates could agree without conflict.

Repair: every occurrence of every recognized alias is independently checked with exact type identity before value comparison. Wrong types add `record_fact_invalid_type:<group>` and block verification.

### B2: malformed recognized containers were ignored

The old container enumerator only included Mapping-shaped values and silently skipped a present string, array, number, boolean, or null container.

Repair: every present `canonical_serialization`, `canonical_authorization`, and nested `verification` key is shape-checked. A non-object emits `recognized_container_not_object:<path>` and blocks verification.

### B3: validation errors did not universally gate evidence

The old evidence expression was not conditioned on an empty blocking-error set. A record with Base64 character count but no Base64 could carry `expected_base64_missing` while still reporting verified evidence.

Repair: `authorization_evidence_verified` now requires a valid serialization profile, all expected values to match, and `not errors`. All current verifier validation errors are blocking.

### B4: hexadecimal comparison was inconsistent

SHA-256 duplicates had some case-insensitive handling, but selected `decoded_sha256` and `last_byte_hex` comparisons were case-sensitive. Syntax validation was not applied uniformly before normalization.

Repair: only recognized hex groups with valid exact syntax are compared case-insensitively. Invalid, truncated, overlong, prefixed, padded, or nonhex values remain rejected.

## 9. Blocking Error Gating Model

Authorization evidence can now verify only when all of these conditions hold:

1. exact-byte serialization profile is valid;
2. required expected values are present, correctly typed, semantically valid, and match;
3. canonical text matches byte-for-byte when present;
4. every recognized serialization fact matches its locally derived fact;
5. all recognized representations are noncontradictory;
6. every recognized occurrence has the exact type and valid value;
7. every present recognized container has object shape;

## 10. Hexadecimal Validation And Comparison

- `sha256` and `decoded_sha256`: exact string type, exactly 64 `[0-9A-Fa-f]` characters, then lowercase-normalized for comparison.
- `last_byte_hex`: exact string type, exactly two `[0-9A-Fa-f]` characters, then lowercase-normalized for comparison.
- uppercase and lowercase encodings of the same valid hexadecimal value are equivalent.
- malformed, nonhex, truncated, overlong, whitespace-padded, and `0x`-prefixed values are rejected before comparison.
- no arbitrary nonhex string group is case-normalized.

Verified compatibility cases include uppercase SHA-256, uppercase decoded SHA-256, and `last_byte_hex=2e` against derived `2E`.
